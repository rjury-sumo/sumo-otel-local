### Live-testing fixes — 2026-07-01

Found while running `./sumo-otel-local.sh -i` end-to-end (Podman on macOS).

- [x] **Spurious `Error: command failed (exit 44) ... Nothing has been changed` printed before every credential prompt on a fresh run** — `sumo-otel-local.sh:secret_get; install_sumo (ACCESS_ID/ACCESS_KEY lookups)` _(bug, small)_
  **Symptom:** On the first run (no stored creds), the ERR-trap report fired twice — once before "Access ID not found" and once before "Access Key not found" — even though "not found" is the normal path. Root cause: `secret_get` returned the backend's non-zero status (`security` exits 44 = `errSecItemNotFound`), and because it's captured with `$(...)` under `set -E`, that non-zero tripped the ERR trap **inside the command-substitution subshell** — the `if !` at the call site can't reach in to exempt it (the same hazard the `endpoints` jq pipeline documents).
  **Done (2026-07-01):** `secret_get` now **always returns 0** and signals "not stored" via empty stdout; each backend lookup is made errexit-exempt (`... || true`). Call sites detect not-found with `[[ -z "$ACCESS_ID" ]]`. Verified empirically on Bash 3.2.57 that this is the only structure that stays clean (`return 1` and `|| rc=$?` both still trip the trap). Added 3 `secret_get` unit tests + 1 end-to-end `install_sumo` flow test; mutation-proven (reverting `secret_get` fails both regression tests). Lint + 157 bats green.
- [x] **A stray stdin line became the cluster name → `--kube-context "kind-Have a question, bug, or feature request?..."` → "context does not exist"** — `sumo-otel-local.sh:init_cluster, install_sumo, reinstall, endpoints, forward` _(bug, medium)_
  **Symptom:** Type-ahead/pasted text buffered during the long `brew`/`kind create` steps was consumed by later prompts; the cluster-name `ask` read kind's banner line, producing a bogus `kind-<junk>` kube context that only failed much later in `helm`. No validation rejected it.
  **Done (2026-07-01):** Added `valid_cluster_name` (RFC-1123-ish: `^[a-z0-9][a-z0-9.-]*$`) and an `ask_cluster_name` wrapper that re-prompts on an invalid name (interactive) and returns the always-valid default unread under `-y`/EOF. Routed all five cluster-name prompts through it. Added 3 unit tests (truth table + re-prompt + unattended); mutation-proven. Lint + bats green. (`confirm_destructive`'s type-the-name gate is intentionally left as-is — different, self-validating semantics.)
- [x] **Install hangs on `helm --wait` when the `sumo-setup` job fails 401 ("Credential could not be verified"); no Sumo region/endpoint support** — `sumo-otel-local.sh:install_sumo; setup job SUMOLOGIC_BASE_URL <- sumologic.endpoint` _(bug + feature, large)_
  **Symptom:** With valid creds for a non-`us1` deployment (or a bad key), the chart's Terraform setup job authenticates against the wrong region and gets HTTP 401; `helm --wait --timeout 10m` then blocks on the CrashLooping pod for the full window. The script only ever sent `accessId`/`accessKey` — it never set `sumologic.endpoint`, so the setup job's `SUMOLOGIC_BASE_URL` defaulted to `us1`.
  **Done (2026-07-01):** Added region/endpoint support + a fail-fast credential pre-flight check. Helpers: `is_known_region` (gates an explicit region against the known deployment list so an alphanumeric typo like `us22` can't build a garbage endpoint), `credential_is_clean` (rejects control-char creds — see hardening below), `sumo_region_to_endpoint` / `endpoint_for_input` (region code or URL → `https://api.<region>.sumologic.com/api/v1`; the latter is the offline/no-network resolver shared by `output` and `--dry-run`), `sumo_api_status` (probe one endpoint; creds via `curl -K -` stdin config, **never** on argv; pipeline errexit-exempt so it can't trip the ERR trap; `--connect-timeout 5 --max-time 10`), and `resolve_sumo_endpoint` (validate a given region/URL, or **auto-detect across all deployments** — handles any region without pinning; called directly, not via `$(...)`, so its reject-`return 1` doesn't trip the trap). `install_sumo` prompts for the region (env `SUMOLOGIC_ENDPOINT`, blank = auto-detect), validates before install, passes `--set-string sumologic.endpoint=<resolved>` to helm, and on a definitive 401 **aborts with guidance** (offers to drop the rejected stored creds) instead of hanging. Added `SUMO_SKIP_CRED_CHECK` (offline/firewalled escape hatch) and env-overridable `HELM_WAIT_TIMEOUT` (was hard-coded `10m`); `output` reflects `SUMOLOGIC_ENDPOINT` via the same offline resolver so it can't drift from install.
  **Adversarial review (2026-07-01, read-only workflow, 14 findings triaged):** fixed a **HIGH curl-config injection** (a newline in a stored/env credential broke out of the `user="…"` line and injected curl options — SSRF / file-write / defeating the 401 abort; live-reproduced) by rejecting control-char credentials up front (`credential_is_clean`, so `yaml_escape` stays sufficient); fixed **region-typo → garbage-URL → re-hang** (`is_known_region` gating); fixed a **false-abort** when the home region is firewalled but another region is reachable+401 (track `saw_unreachable`; abort only when ALL regions answer 401); fixed **install/output drift** for unknown regions (shared `endpoint_for_input`); added an **http:// cleartext warning**; corrected the env-backend "credentials removed" message; and repaired a **HIGH test gap** where `--dry-run` could hit the real network (now stubs curl + asserts it's never called) plus two non-load-bearing assertions (constants defaults; creds-via-stdin now proves creds ARE sent via stdin, not just absent from argv). 176 bats green; the injection guard, region gating, partial-firewall logic, and http warning are all mutation-proven load-bearing; lint + shfmt + yamllint clean.
